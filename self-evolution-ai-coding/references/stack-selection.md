# 技术栈选择与稳定策略

## 原则

不要把技术栈写死在全局 skill 中。全局 skill 只提供选择方法；每个项目必须在 `TECH_STACK_DECISION.md` 固化自己的选择。

## 选择维度

先判断：

- 产品形态：Web、H5、小程序、iOS、Android、桌面、后端服务、AI 工作流。
- 团队能力：团队熟悉语言、招聘难度、维护能力。
- 商业风险：支付、身份、数据安全、长任务、并发规模。
- 生命周期：试验原型、商业 MVP、长期 SaaS、平台底座。
- 生态成熟度：官方维护、社区活跃、测试工具、部署能力。
- AI coding 友好度：类型系统、目录约定、测试生态、错误可定位性。

## 推荐方式

输出项目技术栈时必须包含：

- 选择结论。
- 为什么适合当前项目。
- 不选择其他方案的原因。
- 版本范围。
- 升级策略。
- 禁止漂移规则。
- 验收命令。

## 常见建议

- Web/管理后台：优先 React + Next.js 或 React + Vite。
- 小程序：优先 Taro + React；不要让小程序框架拖累复杂 PC 工作台。
- iOS/Android：优先 Expo React Native；复杂原生能力再评估 Development Build。
- 桌面：优先 Tauri + Web 技术栈。
- 后端：长期商业化优先明确领域边界、类型模型、迁移工具和测试工具，而不是只看框架热度。

## 禁止

- 为单个页面引入新 UI 框架。
- 为单个功能引入新状态管理库。
- 大版本升级混入业务重构。
- 临时 demo 技术进入主项目。
- 前端绕过后端直接访问核心商业服务。
