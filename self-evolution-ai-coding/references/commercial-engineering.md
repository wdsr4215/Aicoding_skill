# 商业化工程编码标准

## 目标

代码必须能被公司员工接手维护。AI 生成代码的目标不是“能跑”，而是“可读、可改、可测、可扩展、可上线”。

## 架构标准

推荐默认结构：

```text
interface / route / controller
  -> use case / service
  -> policy / validator
  -> repository
  -> model / database
  -> provider adapter
```

规则：

- 一个业务能力只有一个 owner。
- 微服务先是边界，后是部署形态。
- 先做模块化单体，再按服务边界抽离部署。
- API route 只处理协议、鉴权上下文、请求响应。
- service 只协调一个业务动作。
- repository 只负责数据访问。
- policy / validator 只负责拒绝和规则判断。
- adapter 只负责外部 SDK、模型 provider、支付 provider、对象存储等。

## 可维护性标准

- 文件过大时优先抽取边界，不继续堆功能。
- 业务规则必须能通过文件名和目录看出 owner。
- 新文件必须出现在项目地图、路由、测试、导出或明确的未来阶段说明里。
- 注释解释“为什么”，不要复述代码“做了什么”。
- 公司团队主要语言是什么，注释和开发说明就优先使用什么语言。

## 商业安全标准

高风险能力必须额外检查：

- 支付金额不能信任前端。
- 权益、额度、余额、订单状态不能信任前端。
- 用户归属、管理员权限、资源 owner 必须在后端校验。
- 写操作必须考虑幂等、重试、并发和回滚。
- 长任务不能阻塞请求线程。
- 外部 provider 调用必须有超时、错误分类和降级策略。

## 验收信号

完成前必须能回答：

- 改哪个服务能改这个逻辑？
- 这个服务是否有唯一 owner？
- 换成独立部署时公共契约是什么？
- 数据库变更是否有迁移和回滚？
- 人工接手时能否从项目地图找到入口？
- 测试或回测是否覆盖主路径和失败路径？
