# 能力迁移可靠性参考

## 使用场景

当任务涉及旧项目能力迁移、AI 入口改造、数字生命/Agent 运行时、多层记忆、Prompt 到 Skill、模型池、工具、评估、进化、扣费或旧数据库兼容时，先读本文件。

目标不是把旧逻辑简单搬到新目录，而是把能力迁移到唯一 owner，并留下可验证、可回滚、可继续微服务化的证据链。本文件是通用工程规则，不绑定某个具体业务项目；具体业务名只应作为本地项目地图里的例子。

## 迁移闭环

每次迁移按这个顺序推进：

```text
能力定义
  -> owner 归属
  -> 旧入口盘点
  -> 行为基线
  -> 新链路接入
  -> Trace 与证据
  -> 守卫测试
  -> 旧入口收口
  -> 地图和账本更新
```

### 1. 能力定义

先写清楚“这是什么能力”，避免把子步骤误建成独立系统。

通用判断示例：

- 候选项生成通常是某个主任务的轻量子产物，不应因为名字独立就单独创建运行单元。
- 分类、匹配、格式修复这类功能通常是工具型 Skill，可以关闭默认进化，但仍应走轻量 Trace。
- 资产提取、实体抽取、依赖检查通常是功能工具型能力，重点是契约稳定、可审计和可重跑。
- 情绪曲线、节拍拆分、路径规划、价格估算、风控打分等领域算法属于业务算法，不属于通用底座算法。

### 2. 唯一 Owner

一个能力只能有一个 owner。

推荐边界：

- API route：协议、鉴权、请求响应。
- business service：业务动作、扣费证据、幂等、产物写入。
- policy / validator：拒绝规则、权限、风险判断。
- repository：数据库读写。
- provider adapter：模型、支付、对象存储、MCP、外部 API。
- foundation / runtime：通用运行、记忆、Skill、模型、工具、评估、进化。

禁止：

- API 直接读 Prompt、拼 prompt、调模型 SDK 或写复杂 SQL。
- 业务层直接绕过统一扣费、模型池、Trace 或记忆 owner。
- 为了迁移临时新增一个平行 service，但旧 service 继续被公开入口调用。

### 3. 旧入口盘点

迁移前至少搜索和记录：

- 公开 API 入口。
- 前端调用点。
- 旧 Prompt / system prompt / template key。
- 旧 AIService / ai_client / provider SDK 调用。
- 数据库写入表、状态字段、幂等键。
- 计费、权益、余额、消费记录。
- 缓存、队列、文件、对象存储和后台任务。
- 管理端入口和可编辑配置。

输出一个当前结论：保留为兼容端口、委托新 owner、删除、或阻断。

### 4. 行为基线

改前先能复述旧行为：

- 输入、输出、错误码和旧响应合同。
- 成功路径和失败路径。
- 是否覆盖写入、追加写入或只读。
- 是否计费，扣什么权益或额度，幂等键是什么。
- 是否持久化，刷新后数据从哪里读取。
- 是否影响旧用户、权益、订单、项目、产物、媒体、资产或业务资源数据。

没有行为基线就不要清旧入口。

### 5. 新链路接入

公开 AI 业务入口迁移后必须进入数字生命、Agent 运行单元，或明确的功能工具型 Skill。

十层证据至少包含：

- user_goal：为什么调用。
- task_spec：任务类型、复杂度、风险、成功标准。
- context：source facts、selected memory、excluded memory、scope。
- skill：主 Skill、引用 Skill、算法引用和版本。
- orchestration：真实执行阶段、已执行和缺失节点。
- tools：模型、MCP、API、文件、外部 provider 调用。
- environment：租户、用户、项目、运行单元、运行深度、权限。
- artifact：真实产物摘要、产物 ID、写入状态。
- evaluation：评分、置信度、扣分原因、命中规则。
- evolution：提案类型、证据、diff、风险、回滚点。

轻量工具型任务可以 L0/L1，但仍要有 trace、产物、契约评估和失败证据。

### 6. Source Fact 与 Memory 边界

业务 source-of-truth 不能被记忆替代。

source facts：

- 业务主数据、项目状态、订单、权益、配置、产物、媒体任务、资产、审计记录。

memory：

- 偏好、经验、关系、历史、世界状态、角色状态、Skill 记忆、唤醒路径。

每次生成要记录：

- 使用了哪些 source facts。
- 唤醒了哪些 memory。
- 排除了哪些 memory，以及为什么排除。
- 记忆 action 是 keep、working only、compress、archive、review、revoke、drop 还是 tombstone。

跨租户、跨用户、跨项目、跨业务对象污染是阻断级问题。

### 7. Skill 与算法融合

算法由代码执行，Skill 记录契约。

业务算法适合留在业务层：

- 领域曲线或评分。
- 内容节奏、流程阶段或任务拆分。
- 媒体、资源或资产拆分。
- 资产统计。
- 计费估算。
- 依赖检查。

Skill 记录：

- 算法名和 owner。
- 输入输出契约。
- 适用边界。
- 版本。
- 评估指标。
- 回滚关系。
- 算法结果 payload key。

Prompt 只接收算法结果和解释，不让模型读自然语言后重新计算。

### 8. 旧库兼容标准

重构只能旁路扩展，不破坏旧库语义。

受保护旧数据：

- 用户。
- 充值。
- 会员 / 订阅 / 权益。
- 额度。
- 订单。
- 项目。
- 业务产物。
- 媒体任务。
- 资产。
- Prompt 资产。
- 模型配置。

允许：

- 新增旁路表。
- 新增索引。
- 新增兼容读取 owner。
- 新增写阻断提示。

禁止：

- 修改旧业务表字段语义。
- 清空旧数据。
- 为了新链路批量改旧用户权益或订单状态。
- 未确认生产库就执行迁移。

未迁移时：

- 读链路降级为空或内置契约。
- 写链路返回明确迁移提示。
- 不允许 500。
- 不允许前端假保存。

### 9. 扣费与平台成本

业务生成扣用户权益或额度，底座评估和进化记平台成本。

业务扣费必须走唯一计费 owner：

- 预检。
- 稳定幂等键。
- 扣费。
- 消费记录。
- 失败回滚或不扣费证据。

底座能力默认不扣用户：

- foundation。
- harness。
- evaluator。
- self_evolution。
- platform diagnostic。

如果一个节点既生成业务产物又触发底座评估，只对业务生成扣费，底座评估成本进入平台预算。

### 10. 守卫测试

清旧前必须有守卫。

推荐测试：

- AI entrypoint guard：公开业务 AI 入口没有旧 Prompt / 旧 AIService 直连。
- internal adapter inventory：内部旧适配器有 owner、风险和清理计划。
- migration compatibility：迁移只新增旁路表，不破坏旧表。
- runtime trace：业务节点写入运行证据。
- memory scope：租户、用户、项目、运行单元隔离。
- managed skill：引用、版本、发布、回滚不假保存。
- evaluator / evolution：真实评分和提案可查。
- model pool routing：绑定只影响当前运行单元。
- billing usage：业务扣费，底座零扣费。

每清一个 owner，至少跑对应守卫和最小业务模拟。

## 清理规则

旧入口短期可以存在，但必须满足：

- 有唯一 owner。
- 只做兼容适配。
- 委托新链路。
- 有清理计划。
- 有守卫测试。

Prompt 只能作为：

- Skill seed。
- Policy seed。
- 任务模板。
- 兼容数据。

Prompt 不能作为：

- 公开运行入口。
- 第二套 Skill。
- 第二套评估器。
- 第二套模型路由。
- 第二套记忆系统。

## 管理端要求

管理端必须展示真实数据，不用说明文字冒充运行结果。

运行管理台应能回答：

- 当前运行单元属于哪个项目和类型。
- 主 Skill 和辅助 Skill 是什么。
- 最近真实 Trace 是哪次。
- 产物是什么，写到哪里。
- 评分多少，为什么扣分。
- 唤醒和排除了什么记忆。
- 触发了什么进化提案。
- 模型池绑定是否只影响当前运行单元。
- 工具、MCP、API 的配置和权限在哪里。

按钮可点击就必须具备 API、审计和旧库兼容；未接入时显示待接入说明，不做假交互。

## 更新文件

迁移完成后按影响范围更新：

- `TOPOLOGY.md`：新增 owner、入口、边界、已知坑点。
- `DEPLOY_MANIFEST.md`：数据库迁移、后端服务、核心路由、环境变量、部署/重启动作。
- 项目错题本：重复错误、错误方向、用户反复指出的问题。
- 全局 reference：多项目通用、稳定、可验收的规则。

不要把流水账写进地图或部署账本。

## 反模式

看到这些信号要停下来重新盘点：

- 一个功能有两个服务都能生成结果。
- 前端状态显示成功，但数据库没有落库。
- 旧 Prompt 页面还能直接影响生产生成。
- 管理端展示 JSON 占位而不是真实 Trace。
- 业务算法被写进 Prompt 让模型自己算。
- 生产迁移没有旧库保护扫描。
- 生成重试变成追加而不是覆盖。
- 只按短 ID 查询，导致跨项目或跨业务对象污染。
- API 层直接扣费或直接扣余额。
- 底座评估扣了用户额度。
