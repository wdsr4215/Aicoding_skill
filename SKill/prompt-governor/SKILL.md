# prompt-governor

## 1. 目标

用于处理提示词、AI 调用、JSON 解析、模型选择、扣费和安全边界。目标是让 AI 能力稳定、可控、低泄露、可计费、可验证。

## 2. 触发条件

- 修改提示词源文件或提示词管理后台。
- 优化随机灵感、灵感扩展、梗概、章节、场次、镜头、资产提取等 AI 产物。
- 排查 JSON 解析失败、输出结构异常、模型耗时过长。
- 排查提示词泄露、输出包含系统指令、输出“如何写”而不是内容本身。
- 调整模型选择、temperature、max_tokens、上下文拼装。
- 涉及余额拦截、扣费、消费记录、异常退款、VIP 拦截。
- 涉及输入长度上限、提示词注入、海量文本攻击。

## 3. 核心原则

1. 提示词必须服务业务产物，不暴露内部写作步骤。
2. 每个 AI 模块职责必须清晰，避免灵感扩展和梗概、章节、镜头重复生成同一层内容。
3. JSON 输出必须有 schema、容错解析、失败提示和重试策略。
4. 长提示词先删重复，再加约束；不要靠堆规则解决质量问题。
5. AI 调用必须检查余额、扣费、消费记录和失败补偿。
6. 输入必须有限长、合法化和注入防护。
7. 模型选择应有默认模型、可用性检查和降级策略。

## 4. 工作流

### Step 1：定位 AI 链路

确认：

- 前端入口。
- 后端 API。
- prompt source。
- prompt template key。
- model config。
- JSON parser。
- billing guard。
- usage log。

### Step 2：判断职责边界

按产物分层：

- 灵感扩展：把输入转化为可写作的故事简介和爆点，不输出写作教程。
- 梗概：建立完整故事主线、冲突、人物、阶段结构。
- 章节：输出正文内容，受目标字数和 max_tokens 约束。
- 场次：把故事拆成可拍摄/可生成视频的场景单位。
- 镜头：输出具体画面、机位、首尾帧和提示词。
- 资产提取：只提取角色、场景、道具、风格资产，不扩写剧情。

### Step 3：质量检查

检查：

- 是否有爆点。
- 是否自然贴合题材。
- 是否避免尴尬缝合。
- 是否避免套话和抽象口号。
- 是否输出目标产物本身。
- 是否符合长度、结构、字段要求。

### Step 4：安全检查

检查：

- 是否可能泄露系统提示词。
- 是否允许用户通过输入覆盖系统指令。
- 是否限制输入长度。
- 是否过滤异常超长文本。
- 是否避免把内部 prompt 原文返回前端。

### Step 5：费用闭环

检查：

- 调用前余额/VIP 权限。
- 调用中模型和 token 记录。
- 调用成功扣费。
- 调用失败不扣费或退款。
- 消费记录可追踪。
- 特殊 AI 调用点的会员/VIP 拦截。

## 5. JSON 解析规则

必须优先使用结构化解析：

- 明确 JSON schema。
- 支持 markdown fence 剥离。
- 支持首尾噪声裁剪。
- 解析失败返回用户可理解错误。
- 记录原始片段用于排查，但不向用户暴露系统提示词。

禁止：

- 靠脆弱字符串切割解析核心字段。
- 解析失败静默吞掉。
- 把模型原始错误直接暴露给普通用户。

## 6. 性能规则

- 单次提示词过长时先拆职责、删重复、压缩示例。
- 不把梗概阶段已经完成的世界观、人物、冲突在灵感扩展阶段重复生成。
- 使用目标字数动态计算 `max_tokens`。
- 对多阶段调用标明每阶段必要性，能合并就合并，不能合并就缓存中间结果。

## 7. 联动规则

必须更新 `TOPOLOGY.md`：

- 新增 AI 模块。
- 修改提示词源路径。
- 修改 prompt key。
- 修改 JSON 解析链路。
- 修复可复用提示词/AI 调用坑。

必须更新 `DEPLOY_MANIFEST.md`：

- 需要同步数据库 prompt template。
- 需要执行 seed/sync 脚本。
- 修改后端 AI service、billing、model config。
- 修改环境变量或模型配置。
