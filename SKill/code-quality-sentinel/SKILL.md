# code-quality-sentinel

## 1. 目标

用于防止巨石文件、重复系统、散装全局状态、幽灵依赖、命名混乱和低质量补丁。目标是让代码保持可读、可测、可演进。

## 2. 触发条件

- 新增功能或重构。
- 单文件过长或职责混杂。
- 重复创建 store、service、API client、modal、配置系统。
- 用户要求代码审查、系统审查、冗余检查。
- 修改跨多个模块、多个端、多个状态源。

## 3. 核心原则

1. 先复用项目已有模式，再新增抽象。
2. 以职责密度判断是否拆分，不机械按行数。
3. UI、状态、API、数据转换、持久化尽量物理隔离。
4. 不引入幽灵依赖；引用必须有 import、依赖必须存在。
5. 命名必须可搜索、可理解、可路由。
6. 不为了架构感过度拆分。

## 4. 拆分阈值

- 超过 400 行：必须评估是否职责混杂。
- 超过 700 行：必须拆分或说明合理性。
- 单文件包含 2 个以上业务域：优先拆分。
- 单组件包含多个 API、多表单、多弹窗、多复杂 effect：优先拆分。

例外：

- 迁移文件。
- 静态配置。
- 生成型类型文件。
- 测试夹具。

## 5. 检查清单

### A. 职责

- 文件是否只承担一个主要职责。
- 是否混合 UI 和复杂业务逻辑。
- 是否混合 API 请求和数据转换。
- 是否混合多个业务域。

### B. 复用

- 是否已有组件/store/service/hook 可复用。
- 是否重复实现 modal、toast、request、auth、billing、model selector。
- 是否引入与项目风格冲突的新库。

### C. 命名

- 目录/文件：优先 `kebab-case`，遵循项目现有规范。
- 组件/类：`PascalCase`。
- 变量/函数：`camelCase`。
- 常量：`UPPER_SNAKE_CASE`。
- 禁止无意义缩写和拼音命名，除非项目已有约定。

### D. 依赖

- import 是否完整。
- package 是否已存在。
- 类型是否来自真实路径。
- 删除文件前是否查引用。

### E. 边界

- 错误态。
- 空态。
- loading。
- 权限不足。
- 余额不足。
- 输入过长。
- 网络失败。
- 并发提交。

## 6. 禁止行为

- 用占位符替代真实代码。
- 随便复制一套全局状态。
- 把所有逻辑塞到页面文件。
- 删除测试来通过构建。
- 大范围重构解决小问题。
- 引入项目不存在的 API 或字段。

## 7. 联动规则

必须更新 `TOPOLOGY.md`：

- 新增共享组件、store、service、hook。
- 删除或合并重复模块。
- 修复可复用架构坑。

如影响部署，联动 `deploy-ledger`。
